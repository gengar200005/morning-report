# ADR-007: UBATP 알림 시스템 폐기 — 장중 알림 불채택

- Status: **Accepted**
- Date: 2026-04-24
- Context session: 2026-04-24 #1
- Supersedes: [plan-001 alert system setup](../plans/001-alert-system-setup.md)
- Related: ADR-005 (entry timing diagnosis)

## 1. 배경

UBATP (장중 Discord 알림) 시스템은 세션 #4 부터 E2E 테스트 이월 중.
구현은 `claude/session-start-UBATP` 브랜치에 완료 (`signals_today.py`,
`notifier.py`). 2026-04-24 ADR-005 완료 후 매매 로직과 알림 설계의 정합성
재검토 결과, **근본적 설계 결함** 확인.

## 2. 판단 근거

### 2.1 현 UBATP 설계의 개념적 오류

현 알림 트리거 = **"오늘 처음 A/B 등급 진입한 종목"** (등급 변화 이벤트).

백테 기반 매매 규칙 = **"미보유 + 쿨다운해제 candidate 중 RS Top N 진입"**
(포지션-aware 신호).

**두 신호는 다른 대상**. 현 알림은:
- 내가 5종목 보유 중이어도 알림 발송 (매수 불가능한데)
- RS 40%대 신규 A등급도 알림 발송 (Top 5 절대 진입 불가)
- 매 스캔마다 여러 건 발송 → 행동 불일치 유발 가능

### 2.2 "올바른 알림" 구현은 기술적으로 가능하나 의미 없음

state-aware Top 5 delta 알림을 만들면 (~2-4시간 구현):
- 162종목 KIS API 현재가 폴링 (초당 20건 rate limit 내)
- 로컬 RS 재계산 + state.json (holdings/cooldown) 조합
- Top 5 변화만 Discord 로

그러나 **장중 RS ≠ 종가 RS**. 14:00 Top 5 와 15:30 종가 확정 Top 5 는
다를 수 있음. 장중 알림 기반 매수 = **"미확정 종가 추정" 에 기반한
투기적 행동** → 백테가 검증한 신호 아님.

**진짜 매매 신호는 종가 확정 후 (15:30+)** 에만 유효. 이 시점 이후
다음 06:00 모닝리포트 생성까지 사용자가 즉시 행동할 수 없는 비대면
시간대이므로, **"종가 후 장중 알림" 과 "다음날 아침 리포트" 의 정보량은
본질적으로 동일**.

### 2.3 청산 알림도 불채택

손절 (-7%) / 트레일링 (-10%) 트리거 알림도 논의했으나 **불채택**:

- 사용자가 **주식앱 기계적 stop-loss/trailing 주문** 으로 처리
- 알림 → 수동 매도 흐름은 시간 지연·감정 개입 리스크
- 주식앱 자동 주문이 알림보다 체결 속도·일관성 모두 우위

## 3. 결정

### 3.1 UBATP 시스템 폐기
- `claude/session-start-UBATP` 브랜치 머지 **하지 않음**
- `signals_today.py`, `notifier.py` 는 해당 브랜치에만 존재. main 에 반영 X
- `docs/plans/001-alert-system-setup.md` → deprecated 마크 (파일 존치,
  의사결정 이력으로 보존)
- Windows Task Scheduler 등록도 하지 않음

### 3.2 운영 구조 확정
```
06:00 KST  모닝리포트 (GitHub Actions)  ← 유일한 매수 신호 생성 시점
          ↓
          사용자 수동 확인 (아침)
          ↓
09:00 시가 근처 매수 주문 (Top 5 - 내 보유 - 쿨다운)
          ↓
보유 중   주식앱 stop-loss -7% + trailing -10% 자동 주문
          ↓
청산 확정 state.json 에 쿨다운 타이머 기록 (60 거래일)
```

### 3.3 CLAUDE.md 반영
- "활성 작업 > 최우선 UBATP E2E" 항목 제거
- 이월 목록에서 UBATP 제거
- 알림 관련 추가 작업 (1주일 알림 모니터링, v2 개선 등) 제거

## 4. 검토한 대안

### 4.1 UBATP 를 재설계하여 Top 5 delta 알림으로 전환
- 기술적으로 2-4h 구현 가능
- 그러나 장중 RS 가 종가 RS 와 다르므로 **장중 알림의 가치 자체가 낮음**
- 구현 비용 대비 실익 없음. **기각**.

### 4.2 장 마감 후 (15:35~16:00) 1회 알림으로 제한
- 종가 확정 후 Top 5 계산 가능
- 그러나 다음 날 아침 모닝리포트와 정보량 동일
- **중복 기능** — 기각

### 4.3 청산 트리거 알림만 유지 (신규 진입 알림 없이)
- 주식앱 자동 stop-loss 가 더 안정적
- 알림 → 수동 매도 경로는 오히려 리스크
- **기각** (사용자 선호 반영)

### 5.3 향후 필요 시 재검토 조건
본 ADR 은 **"2026-04 시점 설계 검토 결과"**. 다음 조건 발생 시 재검토:
- 아침 06:00 리포트가 시스템 장애로 자주 실패 → 백업 알림 필요
- 시장 게이트 급변 등 장중 re-screening 가치가 수치로 입증됨
- 모닝리포트에 포함되지 않는 새 신호 타입 발생 (예: 섹터 rotation
  실시간 감지)

## 6. 교훈

1. **구현 난이도보다 개념적 정합성이 우선** — UBATP 는 기술적으로 어렵지
   않았기에 세션 #4-#8 동안 "일단 만들고 테스트" 방향으로 흘러감. 하지만
   매매 로직 정합성 검토가 먼저였어야 함.
2. **"신호" 용어의 중의성** — 백테의 "신호" (미보유+쿨다운+RS Top N) 와
   알림의 "신호" (등급 변화) 가 같은 단어로 혼재. 이 같은 혼동이 설계
   초기부터 누적되면 "그럴듯하지만 쓸모없는 시스템" 을 만들게 됨.
3. **ADR-005 의 간접 효과** — entry timing 실증이 매매 로직의 구조를
   명확히 드러냈고, 이 명확성이 UBATP 의 결함을 보이게 했음. 원칙 문서화
   가 구현 검토의 전제임.
